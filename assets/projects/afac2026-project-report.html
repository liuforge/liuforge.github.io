2. 赛题与数据特征

AFAC2026 赛题四要求回答金融长文本选择题。题目覆盖五类材料：金融合同、财报、保险条款、监管法规与行业研报。 每类 20 题，共 100 题。题型包含单选、多选和判断类变体；答案提交为 CSV，除逐题答案外还要求 token 统计。

领域题号前缀主要难点适合的处理策略
金融合同fc_a_*条款条件、主体动作、期限金额、否定限制条款六元组、关系槽位覆盖、反证保留
财务报告fin_a_*表格口径、年度/季度、单位换算、同比计算表格行结构化、受限计算 DSL、口径校验
保险条款ins_a_*产品混淆、等待期、责任免除、情景约束产品实体绑定、scenario + exclusion 判断
监管法规reg_a_*应当/不得/无需、例外、期限、主体对象clause matcher、polarity、义务主体比对
行业研报res_a_*图表恢复、指标趋势、公司/行业实体绑定图表文本恢复、数值端点绑定、文档级导航
关键挑战：很多错误不是模型“不会推理”，而是关键证据没有进入最终 evidence ledger，或者表格行/条款上下文被切碎。 因此项目的核心不是堆长上下文，而是让证据短、准、可追踪。

3. 总体架构

项目形成了两条互补路线：一条是 live Agent 机制探索路线，负责检索、证据选择和题型分流实验； 另一条是提交候选路线，负责基于可信官网历史快照进行残差修正与提交复现。

核心 Agent 链路

question -> profiler 判断题型 -> plan_compiler 生成执行计划 -> rewrite_query 按题型和选项生成短 query -> search_bm25 从 KB chunks 检索候选 -> rerank_evidence / qwen_filter_evidence 选择证据 -> read_chunk 扩展局部上下文 -> evidence_ledger 证据片段 atom 化 -> compare_documents / calculate / analyze_clauses -> judge_option 支持 / 反证 / 未发现 -> aggregate_answer 单选/多选聚合 -> validate_answer 格式、引用、token、trace 校验

4. 解决方案分层设计

方案按“可验证子模块”拆解，而不是把所有任务交给一个黑盒 Agent。每个子模块都有明确输入、输出和回归信号： 解析层保证材料结构可用，检索规划层控制召回范围，证据工程层保证证据可追踪，推理工具层只处理适合它的问题， 聚合层负责最终答案风险控制，评估层把错误归因回具体模块。

文档处理层

负责把 PDF/HTML/TXT 转成稳定的结构化语料。核心产物不是纯文本，而是带页码、章节、表格口径、条款号和文档实体的证据块。

检索规划层

先判断题型，再生成短 query 和 must-have 关系槽位。这样可以避免全题统一检索造成候选过宽、上下文过长。

证据工程层

将候选证据整理成支持、反证、未发现三类片段 atom，并记录每个选项缺失的主体、动作、条件、期限或数值端点。

推理工具层

数值比较、条款匹配、保险责任校验优先走确定性函数；Qwen 只处理压缩后的模糊语义判断，降低 token 和漂移。

答案聚合层

单选题按完整性、冲突度、目标文档匹配排序；多选题保留未选原因，避免只看 supported 导致 ABCD 泛化。

评估回归层

银标、badcase、trace 和 token A/B 分开记录。一次修改必须能说明它修的是检索、取证、工具、judge 还是聚合。

5. 为什么选这个架构

5.1 赛题约束决定不能走通用向量 RAG

比赛明确禁止 embedding，且在线模型限制为 Qwen 系列。传统 dense retrieval、向量数据库、非 Qwen reranker 或微调路线都不适合作为主线。 因此检索层选择 BM25F、结构导航、规则化表格/条款抽取，尽量把召回和证据压缩放在确定性系统里完成。

5.2 长上下文不是收益单调的

V2 文档级高召回让正确题数增加，但 token 暴涨导致综合分下降；后续 30K 深证据实验也产生大量答案反转。 这说明“更多上下文”会引入干扰，应该采用渐进式披露：先短证据判断，只有槽位缺失时再补检索。

5.3 多选题需要证据隔离与聚合策略

多选不是逐项支持就结束，还要处理反证、未发现、完整性和选项间冲突。项目把证据从句子升级为片段 atom， 并在聚合层按题型控制 over-select / under-select 风险。

5.4 官网反馈可转化为约束

单次提交只返回总分，但多版本完整答案快照可以形成 0-1 整数约束。当前提交主线正是利用可信历史提交推导“旧答案必错、新答案无下行”的残差候选。

6. 数据流与产物流

6.1 live Agent 数据流

  1. 原始文档解析为文本块、表格行、页章节索引和元数据。
  2. 题目进入 profiler，按财报表格、监管条款、保险情景、研报指标等分流。
  3. 按选项生成 precision query，BM25 检索候选证据。
  4. 证据回填到 relation slots，例如主体、动作、对象、条件、期限、例外、数值端点。
  5. 确定性函数先处理可计算部分；Qwen 只接收压缩后的证据和中间结论。
  6. 答案聚合器生成最终选项，并记录 trace、token、证据片段。

6.2 提交候选数据流

leaderboard_answer_snapshot.csv -> 取已验证历史列作为 baseline -> safe_gain_review_config.json -> residual_candidate_builder -> submission_exporter -> answer.csv / answer_results.jsonl / evidence.json / token_usage.json -> SHA-256 与正式产物校验

6.3 Trace 轨迹展示

trace 不只记录最终答案，而是把每道题从题型识别到检索、取证、工具计算、选项判断和聚合的中间状态全部落盘。 人工 review 时优先看“证据是否覆盖关系槽位”和“候选答案为什么被选/不选”，而不是只看模型最终解释。

1 profiler 识别为财报表格、监管条款、保险情景或研报指标题，决定后续工具链。
2 retrieval 记录 query、top chunks、doc scope、补检索触发条件和召回来源。
3 evidence ledger 把证据整理成片段 atom，保留支持、反证、未发现和缺失槽位。
4 tool results 保存数值比较、条款六元组、保险责任校验等确定性结论。
5 judge_option 逐选项输出 supported / contradicted / not_found 及引用证据。
6 aggregate 单选按完整性和冲突度排序，多选保留未选原因，避免泛化过选。
7 validation 校验答案格式、token 汇总、引用完整性和 qid 覆盖。
8 badcase 将错误归因到检索、取证、工具、judge 或聚合层，形成回归集。

10. live Agent 实验现状与 badcase

live Agent 路线已经能跑通全量 100 题、保存 trace，并支持题型分流、证据账本、确定性函数和 token A/B。 但对当前银标的 full A100 对比显示，某些机制版本仍只有 57/100 exact match，说明它目前更适合作为研发验证平台，而不是提交主线。

实验完成度结果主要问题
mechanism v14 full A100 100/100 57/100 exact match vs silver 监管条款过选、财报表格漏选、保险情景混淆、研报实体绑定不足
deterministic early exit full A100 100/100 无答案漂移，节省 104,101 token 主要节省 judge_option，剩余 token 需针对保险/多跳题局部优化
silver GT 100 行 用户报告约 88-90 题正确 银标不是最终人工 gold，不能把其错误直接反向硬融进 agent
下一步建议：以当前候选作为提交 baseline；live Agent 侧继续做局部机制验证。 每次机制变更先跑固定 regression 子集，至少修复 2 道且旧正确题不回归，再跑 30/100 题全量。

11. 工程化规范

产物规范

  • answer.csv：正式提交文件，含 summary 行与逐题 token。
  • answer_results.jsonl：逐题答案、证据、token、metadata。
  • evidence.json：便于人工 review 的证据展开。
  • token_usage.json：prompt/completion/total 汇总。
  • run_manifest.json：路径、命令、SHA、版本、验收结论。
  • trace.json：完整链路轨迹，用于 badcase 归因。

验收门槛

  • 100 题完整覆盖，不能有 unknown/missing qid。
  • 答案格式符合单选/多选/判断约束。
  • token 总数与逐题相加一致。
  • 候选输出 SHA 必须可复现。
  • 候选答案变化必须有 review config 或 trace 证据。
  • 不把 silver/模型共识直接当作 hidden ground truth。
