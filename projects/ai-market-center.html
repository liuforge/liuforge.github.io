00 项目概览

一句话定义： AI 获客通把“发内容”升级成“内容获客经营”：先用公开内容和行业关键词判断什么值得写，再用 AI 生成可审核内容，发布后把评论/咨询结构化成线索，最后用 BI 看板复盘内容、线索和销售跟进。

业务问题

B2B 服务企业做小红书时，最容易停在“采集很多、生成很多、发布很多”，但有效线索、销售跟进和 ROI 无法证明。

产品解法

用“通用底座 + 行业场景包”拆开共性流程和行业差异，让跨境物流、医美、教育可以复用同一套工作台。

工程目标

前端可演示，后端可联调，数据可落到飞书/BI，关键动作保留人工确认，避免账号风控和合规风险。

技术主线

这个项目的技术核心不是“接一个大模型写文案”，而是把营销获客链路拆成可配置、可回放、可评估的状态机：每个内容、线索和销售动作都有状态、来源和下一步。

环节 技术选择 卡点 处理方式
行业适配 Scenario Pack 不同行业关键词、风险词、线索字段差异大 把关键词、模板、禁用词、CRM 字段都配置化
内容生成 AI Copywriter + 风险词防火墙 营销文案容易夸大承诺或同质化 按能力型、干货型、痛点型模板生成，并高亮敏感词
发布链路 半自动发布助手 全自动发布有账号封禁和平台协议风险 复制标题正文，由销售人工确认发布并回填链接
线索经营 AI 线索解析 + CRM 状态 评论文本不结构化，销售无法快速分配 抽取联系方式、需求、预算、地区、紧急度和跟进状态
效果复盘 Dashboard Aggregation + Feishu BI 曝光点赞不能直接解释成交贡献 聚合内容曝光、互动、咨询、有效线索、成交漏斗

为什么要做场景包

如果把行业知识写死在页面里，系统只能服务一个客户；场景包把客户画像、关键词、模板、风险词和线索字段抽出来，后续替换行业时不用重写主链路。

为什么保留人工确认

小红书等平台对自动化发布和违规导流敏感；AI 负责生成、整理、建议和提醒，人负责审核、发布和关键销售动作，这样演示闭环和账号安全可以同时成立。

数据闭环

内容发布后产生曝光、点赞、评论和咨询，咨询进入线索池，线索解析后绑定销售跟进，最终反哺 BI 漏斗和文案方向贡献。

运营闭环

运营先从竞品内容里选题，再交给 AI 生成内容；销售审核并发布，客户咨询后系统自动整理关键信息，管理者根据 BI 判断哪类内容更有效。

03 前端工程

App Router 页面

工作台、场景配置、看同行、AI 写内容、内容管理、线索管理、BI 看板、飞书 H5 都是独立路由。

Context 状态层

AppStateContext 先充当前端模拟数据库，同时把读写动作封装成稳定接口，后续接 API 时页面不用大改。

API 优先 / Mock 兜底

页面加载先请求后端；后端不可用时保留本地 mock 演示，因此静态站也能完整走主要交互。

当前挂载在个人站的 Demo 是 Next.js 静态导出版本，适合展示产品流、交互和信息架构；真实联调时可切回后端 API，使用 FastAPI/数据库承载持久化。

05 技术选型考量

为什么 Next.js

项目需要同时展示后台工作台、移动 H5、图表和多路交互，Next.js App Router + React Context 足够快，静态导出后也方便作为客户演示链接传播。

为什么 FastAPI

后端以业务接口和数据契约为主，FastAPI + Pydantic 适合快速给前端提供稳定 schema，同时 Swagger 能让联调和客户技术沟通更直观。

为什么不用全自动发布

自动化发布虽然炫，但平台账号风险高；半自动发布把标题正文复制、发布链接回填、指标记录自动化，保留人最后确认。

为什么接飞书

获客项目需要运营、销售和管理者共享进度；飞书多维表/BI 看板是轻量协作层，比一上来做完整 CRM 更适合 MVP 验证。

06 风险与卡点

风险 表现 缓解策略
账号风控 采集/发布自动化触发平台限制 生产主链路采用公开内容/人工导入/半自动发布
内容合规 AI 文案夸大承诺、触碰风险词 场景包风险词、人工审核、敏感词替换
线索质量 曝光高但销售无感 看有效线索、报价、成交，而不是只看点赞收藏
行业迁移 物流规则不能直接套到医美/教育 关键词、模板、线索字段和风险词都配置化
演示与生产差距 静态 demo 没有持久化后端 前端保留 mock 演示，生产联调切 FastAPI + DB
