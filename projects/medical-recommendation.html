00 执行摘要

一句话定义 这个项目不是单纯生成“可能用药”，而是把中文病历结构化、医学知识检索、推理链解释和安全门控串成闭环，让推荐结果可以被验证、被追踪、被迭代。

先收口

中文病历、诊断、检验指标、过敏史、既往用药和出院场景，先转成模型可读的医学决策上下文，但输出必须被候选集和规则门槛收住。

先证据

模型按“诊断理解、药物匹配、安全检查、处方输出”四段式推理，目的是让每个结论都能回到证据，而不是回到语言流畅度。

先闭环

给出候选药物列表、推荐理由、置信度和安全提示，并把低置信样本送入人工复核与 badcase 池。

技术主线

这个系统的关键不是“会不会推荐”，而是“会不会在复杂共病、清单外药物和相互作用风险面前收住”。因此技术栈的每一层都在回答：如何约束、如何解释、如何回流。

为什么先约束再生成

RAG 不只是召回知识，而是把疾病、适应症、禁忌症、药物相互作用和中成药别名同时压进证据链；四段式 CoT 负责把“诊断理解 -> 用药匹配 -> 安全审查 -> 处方输出”拆成可复核步骤。

为什么先对齐再微调

GLM-4-9B 使用 4-bit NF4 + LoRA(rank=16, alpha=32) 进行微调，训练集 3602 条，验证集 10%，cutoff 14k、generation 6k，3 epochs、lr 5e-4、batch 1、grad accum 8。

为什么要把安全做成门槛

SwanLab 跟踪训练漂移，推理显存约 6GB，首 token 延迟 <300ms，吞吐提升 2-3x；候选输出始终被 651 药物白名单和 0.6 置信度门控卡住。

环节 技术选择 卡点 处理方式
证据召回 Neo4j + FAISS/Milvus + BGE-M3/M3E + reranker 检索噪声和别名覆盖不均衡 按病历字段定向召回，再做重排和白名单过滤
推理生成 GLM-4-9B + CoT 蒸馏 链路漂亮但可能偏离医学事实 把每一步绑定到证据和规则
偏好对齐 DPO chosen/rejected 清单外药物和中成药幻觉 badcase 回流做偏好对齐
安全门控 置信度阈值 + DDI + 白名单 阈值过高会误杀，过低会放险药 人工复核队列校准阈值

最危险的不是答错

医疗推荐里最危险的错误是“看起来合理但清单外、禁忌、相互作用或特殊人群不适用”。所以系统不能只优化 AVG_P / F1，还要把清单外药物、低置信答案、冲突证据和安全告警单独拉出来看。

为什么保留人工复核

人工复核不是为了替代模型，而是给阈值、DPO 负样本和知识库缺口提供校准信号。低置信样本进入复核队列后，可以反向更新别名、禁忌规则、提示词和偏好数据，形成真正的医疗安全闭环。

这套系统的叙事重点是“把生成能力关进医学边界里”：RAG 给证据，LoRA 让模型贴合任务，CoT 让理由可检查，安全门控决定哪些答案不能交出去。

01 问题与约束

难点 具体表现 系统策略
代谢疾病共病复杂 糖尿病、甲状腺、血脂、肝肾功能等信息交织，单个诊断无法决定用药。 先做结构化抽取，再把诊断、指标、禁忌和既往用药一起送入推理。
651 种候选药物 输出空间大，且中西药命名、别名和适应症覆盖不均衡。 候选药物白名单、别名规范化、RAG 补足说明书和指南知识。
中成药幻觉 模型容易生成清单外药物或把相似功效药混淆。 候选集约束、置信度阈值、规则校验和 DPO 偏好数据回流。
医疗安全要求高 推荐不仅要命中，还要解释为什么可用、为什么不能用。 加入禁忌症、相互作用、特殊人群和低置信拒答门控。

系统架构图

数据层

原始中文病历 JSON、651 种候选药物清单、疾病指南、药品说明书、文献摘要和人工整理知识共同构成输入。训练集约 3602 条，验证集占 10%，用于做推荐指标与格式稳定性评估。

模型层

基座使用 GLM-4-9B，采用 4-bit NF4 量化和 LoRA 训练约 5000 万参数，冻结主体模型参数，在有限 GPU 资源下保留长上下文能力。

03 RAG 知识增强

Knowledge Graph

用 Neo4j 组织疾病、症状、检验指标、药品、适应症、禁忌症和相互作用，让系统能做结构化安全检查。

Vector Retrieval

用 BGE-M3 / M3E 向量化指南、说明书和文献片段，FAISS 或 Milvus 负责召回，BGE-Reranker 做二次排序。

Prompt Packing

只把与当前病历诊断、指标异常和候选药物相关的知识打包进上下文，避免长文本噪声稀释关键证据。

RAG 在这里的角色不是“让模型知道更多”，而是把模型的每个推荐约束到可追溯证据上。后续评估需要单独统计检索命中、证据使用和最终推荐命中，不能只看最终 F1。

04 微调与训练配置

模块 配置 作用
Base Model GLM-4-9B 承担中文医疗长文本理解与推荐生成。
Quantization 4-bit NF4 降低显存占用，PPT 记录推理显存约 6GB。
LoRA rank=16, alpha=32 只训练少量适配参数，保持主模型稳定。
Context cutoff 14k, generation 6k 容纳长病历、检索证据和推理链。
Training 3 epochs, lr 5e-4, batch 1, grad accum 8 在小样本竞赛数据上做稳态 SFT。
Observation SwanLab 记录 loss、验证指标、样本输出和训练漂移。

05 推理链蒸馏与偏好对齐

DeepSeek-R1 CoT 蒸馏

用强推理模型生成 few-shot 思维链，把“诊断判断、药物匹配、安全审查、处方输出”拆成可读步骤，再蒸馏给较小模型，提升解释性和复杂共病处理能力。

DPO Badcase 对齐

把“清单外药物、错误中成药、忽略禁忌、格式不合法”等样本构造成 chosen / rejected 对，后续用于 DPO 或规则偏好训练，让模型明确学习什么答案不可接受。

诊断理解 -> 匹配适应症 -> 检查禁忌/相互作用 -> 输出候选药物与理由
低置信度或安全冲突 -> 进入人工复核队列，而不是强行生成处方

06 安全门控

白名单

输出药物必须命中 651 候选清单或规范别名。

置信度

PPT 中设定阈值为 0.6，低于阈值触发保守输出。

DDI

检测药物相互作用、禁忌症和重复用药风险。

审计

保存病历摘要、证据片段、推理链和最终药物列表。

本页描述的是竞赛与工程原型系统，不构成医疗建议。真实临床落地必须经过医生复核、伦理审查和持续安全验证。

数据流水线与验证闭环

闭环指标 为什么需要 回流动作
AVG_P / F1 / Top-K 衡量推荐列表与 gold medication 的匹配程度。 定位召回不足、排序错误或共病理解错误。
格式合法率 医疗推荐必须稳定输出可解析列表。 加入格式修复样本和解码约束。
清单外幻觉率 直接衡量模型是否生成候选集外药物。 强化白名单校验和 rejected DPO 样本。
禁忌/相互作用告警率 不能只追求命中率，需要识别风险输出。 更新知识图谱、规则库和安全提示模板。
低置信复核率 用人工审核承接模型不确定样本。 沉淀成高价值标注样本，进入下一轮训练。

补齐后的项目闭环是：每次预测都保留“病历摘要、检索证据、推理链、规则检查、最终列表、gold 对比和人工判断”。这样可以把一次竞赛模型扩展成可持续迭代的医疗推荐系统原型。

08 风险与下一步

RAG 覆盖不足

Knowledge Gap

指南、说明书和真实临床路径覆盖不完整时，模型会把语言模式误当医学证据。下一步需要补充高质量来源，并在评测中拆分检索召回和答案正确性。

中成药与别名混淆

Alias Risk

中成药功效描述相近，且别名、商品名、通用名混用。需要药品别名库、候选集归一化和清单外强拒绝共同处理。

过度依赖 CoT

Reasoning Risk

推理链看起来合理不等于推荐正确。需要把每一步绑定到检索证据和规则检查，而不是只评估最终文本流畅度。

临床落地边界

Clinical Boundary

真实部署前必须引入医生审核、审计日志、版本化规则库和灰度验证。本项目阶段定位为竞赛系统与工程原型。

09 技术选型与卡点

选择 为什么这么选 卡点 回路
GLM-4-9B + 4-bit LoRA 在小团队和有限显存下保留中文长文本能力。 容易过拟合小样本、长上下文下会漂移。 用验证集、SwanLab 和 badcase 回流控制。
RAG + 知识图谱 让推荐依赖外部医学证据而不是只靠参数记忆。 检索噪声、证据切片和别名覆盖不足。 按病历字段定向检索，并将漏召回样本回填知识库。
CoT 蒸馏 + DPO 把医学推理过程拆成可解释链条，并约束错误答案。 思维链漂亮但可能不正确。 把思维链绑定到证据与安全规则，错误样本做偏好对。
置信度门控 医疗场景宁可保守，也不能输出不安全推荐。 阈值过高会误杀可用药物。 用人工复核队列动态校准阈值。

这个项目最难的不是把模型跑起来，而是把“推荐正确”和“推荐安全”同时做成闭环。只要一个环节失真，后面再好的生成能力都会被安全门挡住。
