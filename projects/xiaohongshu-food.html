00 执行摘要

一句话定义 这个项目把“搜小红书帖子”拆成可观测的推荐决策链：先理解用户要找什么，再渐进式找候选内容，随后识别营销噪声、抽取真实 POI，最后把用户反馈和线上 trace 回流到策略、提示词和缓存。

先有判断

系统不是为了搜得更多，而是先判断用户真正要找什么、什么帖子值得进入推荐池。

先保可信

热门推荐、POI 补全和缓存可以异步更新，但最终结果必须把营销噪声、重复帖子和过期 POI 挡在外面。

先做闭环

签名失败、爬取超时、LLM 失败、缓存击穿都必须能被观测、被归因、被回流。

技术主线

这条线的本质不是搜索链路本身，而是如何把“可信推荐”做成一个可解释、可降级、可回流的判断系统。下面三块分别说明为什么要这样设计，而不是只说明系统如何运转。

为什么先判意图

IntentParser 用 bge-small-zh-v1.5 384d 向量 + HNSW 做意图召回，cosine 阈值 0.85；它的作用不是多找，而是先把搜索意图收窄到值得继续展开的范围。

为什么要分阶段扩候选

Spider 用四阶段渐进式搜索扩候选池，Analyzer 用 LLM JSON schema 判断内容可信度，POIEnricher 负责去重和地点归一；这里的重点不是“跑完四步”，而是每一步都在收紧可信边界。

为什么要有可回流的结果

FastAPI + JWT + SSE 承载长链路，Token Bucket 限流 5 QPS、burst 10；L0 LRU、L1 Redis、L2 PostgreSQL/pgvector 形成缓存与持久化层，Circuit Breaker 和 K8s/HPA 兜住高峰和故障，但这些工程手段最终都要服务于可信推荐和可回流评测。

环节 技术选择 卡点 处理方式
意图解析 向量检索 + LLM rerank 长尾意图和跨城市表达 Miss 才调用 rerank，并缓存短期结果
内容搜索 渐进式 Spider 反爬、签名、候选不足 分阶段扩候选，达到 15 条即停
内容分析 端到端 LLM + schema 广告噪声、营销口吻、重复帖子 confidence < 0.5 直接丢弃并回流标注
服务承载 SSE + 缓存 + 降级 长耗时和尾延迟 把状态拆成可见事件流并做多级缓存

什么算推荐成立

不是返回一堆热帖就成立，而是用户意图、城市/商圈、菜系场景、真实 POI、帖子可信度和推荐理由都能对齐。只要其中一个环节没有证据，结果就应该降权、补检索或进入回流。

哪些失败必须回流

403/429、候选不足、重复店名、营销软文误判、POI 地址冲突、缓存过期和 LLM JSON 解析失败都不是“偶发错误”。它们会直接改变推荐质量，因此必须进入 trace 和指标面板，而不是只写进日志。

这个项目的工程价值在于把“内容平台的不确定性”拆成可降级的服务边界：能缓存的缓存，能提前判断的提前判断，必须实时调用的部分才交给 Agent 链路。

系统架构图

Request Layer

FastAPI 负责 API 网关，JWT 处理身份，Token Bucket 对高风险接口限流，SSE 让长链路任务以事件流形式展示进度。

Storage Layer

L0 LRU、L1 Redis、L2 PostgreSQL + pgvector 组成三级缓存与持久化层，兼顾毫秒级读取和可追溯历史。

架构主线是“延迟与一致性取舍”：推荐系统不要求每次立即拿到全量最新信息，但要求每个阶段都有状态、失败可降级、结果可解释。

02 IntentParser

方案 延迟 准确率 取舍
纯 LLM 1-2s 约 95% 准确但成本和延迟较高。
纯规则 <10ms 约 60% 快但泛化能力不足。
向量 + LLM rerank 命中 <100ms，未命中 1-2s 约 93% 作为线上默认策略，兼顾速度和泛化。

Embedding

使用 BAAI/bge-small-zh-v1.5 的 384 维向量表达用户意图、城市、菜系和场景。

Retrieval

HNSW 近似检索把意图匹配复杂度降到 O(log n)，cosine 阈值设为 0.85。

Miss Handling

向量未命中时才调用 LLM rerank，并把新模式写入短期缓存，TTL 约 1 小时。

反爬适配

PPT 中方案包含 HMAC-SHA256 签名、Playwright 设备指纹、随机等待和 IP 池。项目页保留为技术方案，同时在闭环中加入 403/429、签名失败和候选不足的合规风险门控。

候选池策略

不一次性把所有关键词打满，而是按“精确、本地、扩展、长尾、兜底”逐步扩大搜索范围。候选池达到 15 条后提前停止，减少噪声和接口成本。

04 Analyzer 与 POIEnricher

内容可信度分析

Analyzer 用端到端 LLM 判断帖子是否真实有用，过滤广告口吻、模板化夸张、无细节种草和重复营销内容，并输出可解析 JSON。

POI 补全

POIEnricher 把帖子中的店名、地址、商圈和菜品信息映射到真实地点，合并重复餐厅，降低“同店多贴”导致的排序膨胀。

策略 PPT 记录收益 闭环补充
端到端 LLM 分析 开发周期 2 周降到 2 天，准确率 62% 提升到 89%。 需要离线标注集校验真实广告过滤和推荐准确性。
CoT 分析字段 分析质量提升 8-12%。 只作为解释与调试信号，不直接等同于正确性。
JSON Schema + 重试 提升结构化输出稳定性。 统计 schema pass rate、retry 次数和最终丢弃率。
confidence < 0.5 丢弃 减少低质量推荐。 需要监控误杀率，避免把小众但真实的好店过滤掉。

05 缓存与成本控制

L0 LRU

进程内缓存，命中可达微秒级，适合热意图和短期重复请求。

L1 Redis

跨实例共享缓存，读延迟约毫秒级，是线上主要降本层。

L2 PostgreSQL

长期存储查询、候选、推荐、反馈和 pgvector 向量索引，支撑复盘与训练样本沉淀。

Cache-Aside 加随机 TTL 48-72 分钟用于避免缓存雪崩；但美食 POI 有时效性，热门店铺状态、营业时间和口碑漂移需要通过反馈和定时刷新补偿。

06 稳定性与降级

机制 参数或策略 保护对象
Circuit Breaker 60s window，失败率 50%，最少 10 次调用后触发。 防止单个外部服务拖垮整条推荐链路。
LLM 降级 MiniMax -> GLM-4 -> Kimi -> 缓存热门推荐。 在模型服务故障或限流时保留可用结果。
Token Bucket 5 QPS，burst capacity 10。 保护小红书搜索与高成本接口。
Backpressure 队列长度和超时控制。 避免高峰期堆积导致整体雪崩。

07 云原生部署

Container

多阶段 Docker 构建将镜像从约 1.2GB 压缩到约 180MB，使用 non-root 用户运行，并配置 healthcheck。

Kubernetes

Pod anti-affinity 避免单节点故障，startup / liveness / readiness probes 拆分启动、存活和可接流量状态。

HPA 策略 配置 目的
副本数 min 2, max 20 保证基础可用性并支持突发流量。
扩缩指标 CPU 70%, memory 80% 用资源指标控制横向扩缩容。
稳定窗口 scale-up 60s, scale-down 300s 避免推荐高峰时频繁抖动。

补齐后的闭环不是只看“推荐出来了没有”，而是把每一次推荐拆成可复盘 trace：用户意图、搜索阶段、候选池、分析 JSON、POI 合并、最终排序、用户反馈和后续修正都能被记录。

09 风险与下一步

平台合规风险

Compliance

反爬适配只能作为技术原型讨论，真实部署需要遵守平台条款、控制请求频率，并优先接入合法授权数据源。

营销内容误判

Noise

广告与真实分享边界模糊，需要用人工标注集、用户反馈和复核队列持续校正 Analyzer，而不是只相信 LLM 解释。

POI 时效性

Freshness

餐厅停业、搬迁、改名会让历史推荐失效。需要引入更新时间、外部 POI 校验和用户纠错入口。

成本与尾延迟

Latency

LLM、浏览器自动化和向量检索叠加后容易出现尾延迟。下一步用 trace 看板定位慢阶段，再决定缓存、降级或异步化。

10 技术选型与卡点

选择 为什么这么选 卡点 回路
向量 + LLM rerank 兼顾低延迟和意图泛化，比纯规则更稳。 同义意图、跨城市和长尾菜系容易漏命中。 失败样本回流到短期缓存和 rerank prompt。
SSE + 异步 Agent 长链路任务需要可见的阶段状态，而不是静默等待。 事件粒度过粗会像黑盒，过细会拖慢前端。 按“解析/搜索/分析/补全”四阶段拆分事件。
三级缓存 L0/L1/L2 兼顾热数据速度、跨实例共享和历史追溯。 POI 时效性和缓存过期策略容易失衡。 随机 TTL + 用户纠错 + 定时刷新一起做。
K8s + HPA 比单机部署更能承接突发流量和失败隔离。 服务多了以后成本与观测复杂度上升。 用指标、熔断和降级把复杂度压回可控范围。

这个系统的关键不是“抓到最多帖子”，而是“抓到最可信、最能解释、最不容易误导人的帖子”。因此技术选型一直围绕成本、延迟、质量和可维护性四个目标做权衡。
