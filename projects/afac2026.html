1. 执行摘要

当前工程结论： 不要把 live Agent 机制探索路线与提交候选路线混在一起。提交候选来自“可信历史答案快照 + 残差安全修正”；live Qwen/检索推理路线适合继续做机制研究，但在银标对比上仍需通过 badcase 回归逐步吸收。

2. 赛题与数据特征

赛题四要求回答金融长文本选择题，覆盖金融合同、财报、保险条款、监管法规与行业研报。真正难的不是题量，而是证据口径、条款条件、数值单位和文档实体绑定。

领域 题号前缀 主要难点 处理策略
金融合同 fc_a_* 条款条件、主体动作、期限金额、否定限制 条款六元组、关系槽位覆盖、反证保留
财务报告 fin_a_* 表格口径、年度/季度、单位换算、同比计算 表格行结构化、受限计算 DSL、口径校验
保险条款 ins_a_* 产品混淆、等待期、责任免除、情景约束 产品实体绑定、scenario + exclusion 判断
监管法规 reg_a_* 应当/不得/无需、例外、期限、主体对象 clause matcher、polarity、义务主体比对
行业研报 res_a_* 图表恢复、指标趋势、公司/行业实体绑定 图表文本恢复、数值端点绑定、文档级导航

很多错误不是模型“不会推理”，而是关键证据没有进入最终 evidence ledger，或者表格行/条款上下文被切碎。因此项目核心不是堆长上下文，而是让证据短、准、可追踪。

3. 总体架构

双路线： 一条是 live Agent 机制探索路线，负责检索、证据选择和题型分流实验；另一条是提交候选路线，负责基于可信官网历史快照进行残差修正与提交复现。

文档处理层

把 PDF/HTML/TXT 转成稳定的结构化语料，保留页码、章节、表格口径和条款号。

检索规划层

先判断题型，再生成短 query 和 must-have 关系槽位，避免全题统一检索。

证据工程层

把候选证据整理成支持、反证、未发现三类片段 atom，并记录缺失槽位。

推理工具层

数值比较、条款匹配、保险责任校验优先走确定性函数，Qwen 只处理压缩后的模糊语义判断。

答案聚合层

单选按完整性与冲突度排序，多选保留未选理由，避免 over-select。

评估回归层

银标、badcase、trace 和 token A/B 分开记录，一次修改必须能归因到具体模块。

4. 解决方案分层设计

题型分流

财报走 table/numeric extractor；监管走 clause matcher；保险走 product + scenario + exclusion。

关系级证据覆盖

不是关键词覆盖，而是 subject/action/object/condition/deadline/exception 逐项覆盖。

证据契约

中间态稳定，trace 才能 review，badcase 才能归因。

答案级门禁

单选多 supported 时按完整性和冲突度排序；多选保留未选原因。

受限计算

表格题优先走确定性计算，避免大模型做口径漂移。

回归闭环

错误必须回流到正确层，而不是一股脑塞回训练集。

4.2 按题型分流

题型 优先工具链 模型介入点 防错点
财报表格比较题 表格行抽取 -> 指标/年份/单位绑定 -> 数值比较 只解释计算结论 单位、口径、合并/母公司
监管条款题 条款召回 -> 六元组抽取 -> polarity 比对 处理“除外、仅、无需”等边界 主体、动作、对象、期限不能缺槽
保险情景题 产品绑定 -> scenario 解析 -> exclusion 校验 判断情景是否命中责任或免责条件 等待期、责任免除、合同生效日
金融合同题 条款定位 -> 条件链还原 -> 反证收集 合并跨条款条件并判断过度概括 应当/可以/不得、金额阈值、期限触发
行业研报题 章节导航 -> 图表文本恢复 -> 指标端点绑定 判断趋势描述与图表/段落是否一致 公司实体、行业口径、时间区间

5. 为什么选这个架构

赛题约束决定不能走通用向量 RAG

比赛明确禁止 embedding，且在线模型限制为 Qwen 系列，所以检索层选择 BM25F、结构导航与规则化表格/条款抽取。

长上下文不是收益单调的

V2 的高召回让正确题数增加，但 token 暴涨导致综合分下降，说明“更多上下文”会引入干扰。

多选题需要证据隔离与聚合策略

多选不是逐项支持就结束，还要处理反证、未发现、完整性和选项间冲突。

官网反馈可转化为约束

单次提交只返回总分，但多版本快照可以形成 0-1 整数约束，排除不可能答案。

6.2 Trace 轨迹

1. profiler 识别为财报、监管条款、保险情景或研报指标题，决定后续工具链。

2. retrieval 记录 query、top chunks、doc scope、补检索触发条件和召回来源。

3. evidence ledger 把证据整理成片段 atom，保留支持、反证、未发现和缺失槽位。

4. tool results 保存数值比较、条款六元组、保险责任校验等确定性结论。

5. judge_option 逐选项输出 supported / contradicted / not_found 及引用证据。

6. aggregate 单选按完整性和冲突度排序，多选保留未选原因，避免泛化过选。

7. validation 校验答案格式、token 汇总、引用完整性和 qid 覆盖。

8. badcase 将错误归因到检索、取证、工具、judge 或聚合层，形成回归集。

7. 关键 Trick 与工程经验

Trick 解决的问题 效果 / 注意点
题型分流 所有题走同一 judge 会混淆 按财报、监管、保险、合同、研报分流
关系级 evidence coverage 关键词覆盖不等于关系覆盖 按 subject/action/object/condition/deadline/exception 检查
表格结构化抽取 财报表格行被切碎，年度/单位/口径丢失 抽出 metric/year/value/unit，并用受限计算函数比较
deterministic early exit 很多题已可确定 节省 104,101 token，降幅 11.33%
答案聚合 guard 单选多 supported、多选过选/漏选 按完整性、冲突度、目标文档匹配度排序
Leaderboard constraints 官网只给总分，没有 ground truth 可信提交快照要变成硬约束，避免把语义真值越层当隐藏标签

10. 风险与后续计划

风险 表现 控制策略
过度硬融银标 agent 追候选答案而不是追证据 银标只做回归对比；修改必须落到机制层
证据漂移 候选证据换文档或换口径后答案仍被支持 实体绑定、doc scope、关系槽位覆盖、引用校验
多选过选 supported 过宽导致 ABCD 泛化 反证保留、限制词 guard、单选/多选分聚合
表格口径错误 季度/全年、母公司/合并、单位混淆 bind_metric_year_value + validate_unit_and_scale
token 成本反弹 补检索和二次 judge 全题触发 渐进式披露，只对缺槽 not_found 定向补检索

短期里程碑

  1. 提交或确认当前候选，获得 88/89/90 的官网反馈。
  2. 将反馈作为新硬约束，更新 residual candidate 搜索空间。
  3. live Agent 侧继续做局部机制验证，但不与提交主线混跑。
