00 项目概览

一句话定义 MR-Search 训练的不是单轮回答，而是 search-reason-answer loop。模型要学会什么时候搜、搜什么、怎么用证据、什么时候停，以及如何避免无证据乱答。

Search

生成可检索 query，避免长问题噪声和重复搜索。

Evidence

把 retrieval hit 转成模型真正可用的上下文证据。

Answer

搜到后正确抽取答案，而不是在 final 阶段破坏答案。

当前阶段已经不是“环境是否能跑通”，而是“检索增强如何真正转成 answer correctness”。后续优化不能只看 recall 或 search count，而要拆成 query、retrieval、context、answer、reward 五条链路逐项诊断。

阶段 目的 决策规则
quick32 快速判断方向是否崩坏 只看方向，不做最终决策
quick128 做可比决策 quick32 正确再跑

03 诊断指标

指标 含义 异常时优先排查
final_em / final_f1 最终答案质量 只能说明结果，不能单独解释原因
retrieval_alias_hit_rate 证据中是否包含 gold alias 低：query/retrieval；高但 F1 低：answer extraction
no_hit_and_wrong_count 没搜到且答错 query rewrite、retrieval ranking、recall、evidence selection
hit_but_wrong_count 搜到了但答错 evidence 使用、answer span extraction、final answer prompt
final_worse_after_retry_count retry/reflection 后最终答案变差 reflection loop、stop gate、answer preservation
mean_search_count 平均搜索次数 高但 hit 不高是无效搜索；高且 final_worse 高是过度 search
repeated_query_rate 重复 query 比例 query rewrite、hard stop repeated query、loop gate
finished_rate 是否正常输出 answer format、termination、force final answer、预算耗尽行为

Trace 归因字段

  • Query: raw_search_query, cleaned_query, first_hop_query, search_queries
  • Retrieval: retrieval_docs, retrieval_scores, retrieval_alias_hit, query_gold_hit
  • Context: selected_evidence, evidence_gold_hit, state_tokens, clip_ratio
  • Loop: search_count, repeated_queries, hard_blocks, final_worse_after_retry

判断原则

如果 no_hit_wrong 多，不要主要惩罚 answer；如果 hit_but_wrong 多，不要继续堆 retrieval；如果 final_worse 多，不要鼓励更多 reflection；如果 finished 低，先修 termination。

04 Badcase Playbook

Query Too Broad

raw query 噪声

raw query 太长，混合多个 clue，BM25 被干扰。解决方式是 single-rewrite v2 first-hop 更稳；multi-candidate 只能做 fallback，不应主导排序。

Rewrite Drops Constraint

裁剪丢约束

first-hop hit 提高，但裁掉了关键 disambiguation clue。要保留 raw fallback，但 fallback 只补充低置信场景。

Hit But Wrong

搜到但答错

retrieval_alias_hit=true，但 final answer 错。这不是召回问题，而是 evidence 使用或答案抽取问题。

Candidate Correct Final Wrong

final 破坏答案

candidate/evidence 中有正确答案，final 阶段却改坏。优先做 deterministic extraction 诊断，再改 answer prompt。

No Hit Wrong

没搜到且错

no_hit_and_wrong_count 高，说明 query/retrieval 没把答案带进 context。先改 query rewrite、retrieval ranking、evidence selection。

Over Search / Repeat

过度搜索

mean_search_count、repeated_query_rate 或 final_worse_after_retry 高。hard gate 优先：max search count、repeated query block、retrieval hit 后 answer。

05 Query 与 Retrieval

Query Rewrite 结论

single-rewrite v2 最稳。raw query 太长会把多个 clue 混在一起，BM25 容易返回主题相近但不含答案的文档。

关键补充

multi-candidate / group fusion 可用，但不能把多候选做成无条件并列融合；上下文管理必须独立优化。

策略 问题 结论
raw query 太长、太噪声 只作 fallback
first-hop rewrite 可能裁掉约束 保守保留关键 clue
multi-candidate fusion 容易并列过宽 不作为长训主线

06 Reward Design

保留

  • 主 reward：EM/F1 outcome
  • 轻量 repeated/search penalty
  • hit 后正确使用证据的 diagnostic/reward

避免

  • 不要让 evidence overlap 文本分数主导 reward
  • 不要把“少搜”做成强 reward
  • 不要用 reward 代替 hard gate

当前 reward 方向应回到 outcome 主导。process reward 可以有，但必须轻约束，不能把“少搜”变成主要目标。

07 核心原则

训练不是越久越好

如果 query、context 或 reward 归因错了，继续长训只会把错误行为强化。

召回不是最终目标

真正要优化的是“搜到正确证据”能否稳定转成“最终答案正确”。

搜索控制应优先机制化

重复搜、过度搜、不及时 stop 更适合用 hard gate，而不是强 reward 惩罚。

Reward 要做归因

no-hit wrong、hit-but-wrong、candidate-correct-final-wrong、final-worse-after-retry 应该落到不同修法。

一句话总结：MR-Search 的关键不是继续压缩 search count，而是把“搜到正确证据”稳定转成“最终答案正确”。

下一步计划

  1. 以当前 single-rewrite v2 和 hard gate 作为主线，继续做 fixed regression。
  2. 每次机制变更先跑 30/100 题全量，再看 EM/F1、hit、final_worse、search_count。
  3. badcase 只回流到对应层：检索、query、证据、reward、SOP。
